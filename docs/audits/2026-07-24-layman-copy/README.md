# First-time reader copy audit

Captured before the July 24 plain-language pass.

## Screens reviewed

1. `01-new-investigation.png`, new investigation entry point.
2. `02-report-summary.png`, VVV investigation summary.
3. `03-key-facts.png`, VVV key facts.
4. `04-person-report.png`, person report summary.
5. `05-how-it-works.png`, methodology overview.

## Main problems found

- One idea had several names, including coverage, readiness, evidence status, and snapshot status.
- Safety checks and optional follow-up questions could appear side by side without explaining the difference.
- Internal research terms such as canonical, provenance, reconciliation, liveness, and forensics reached the reader.
- Provider failures exposed technical details before explaining what they meant for the report.
- Market terms such as FDV and CEX were not explained.
- Saved-report language sounded legal or technical instead of simply stating when data was saved.

## Language system applied

- Use **result** for what ARGUS found.
- Use **report status** for whether the result has enough finished checks.
- Use **warnings** for negative findings.
- Use **follow-up questions** for useful research that is not a failed safety check.
- Use **saved report** for a fixed historical report.
- Use **official token** for a token confirmed through the project’s official account or website.
- Keep technical provider details behind a disclosure.
- Pair uncommon market acronyms with a plain explanation.
